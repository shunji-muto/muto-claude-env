# スマートコミット実行

未コミットのファイルを分析し、論理的に関連する変更を適切な粒度でコミットします。

## 実行手順

### 1. 品質チェック（コミット前必須）

プロジェクトで定義されている品質チェックを実行：
- Linter/Formatter (ESLint, Prettier, ktlint, rustfmt 等)
- 型チェック (TypeScript, Flow 等)
- `bun run lint`などプロジェクト固有のコマンドを確認

### 2. 未コミット状況確認

```bash
# 変更されたファイルをリスト
git status

# 変更の統計情報を確認
git diff --stat

# 具体的な変更内容を分析
git diff
```

### 3. 変更内容のグループ化

- 機能追加、バグ修正、リファクタリング、スタイル修正など論理的に分類
- 関連するファイルをまとめてグループ化
- 不要なデバッグコード・コメントの削除

### 4. 適切な粒度でコミット

各グループごとに独立したコミットを作成：
- Conventional Commits 形式でメッセージ作成
- 1 コミット = 1 つの論理的変更の原則

**コミットメッセージ形式:**
```
<type>(<scope>): <subject>

<body>

Co-Authored-By: Claude <noreply@anthropic.com>
```

**コミットタイプ:**
- `feat`: 新機能の追加
- `fix`: バグ修正
- `docs`: ドキュメントの変更
- `style`: コードの見た目の修正（空白、フォーマットなど）
- `refactor`: リファクタリング（機能の変更を伴わないコード改善）
- `test`: テストの追加・修正
- `chore`: ビルド・補助ツール・ライブラリの変更
- `perf`: パフォーマンス改善
- `build`: ビルド関連の変更
- `ci`: CI設定の変更
- `revert`: 以前のコミットの取り消し

### 5. 品質確保

- lefthook プレコミットフックが実行されることを確認
- コミットメッセージ検証が通ることを確認
- 最終的な`git status`でクリーンな状態を確認

### 6. プッシュ（要求された場合）

ユーザーが「プッシュ」「push」を含む指示をした場合のみ実行：

```bash
git push origin HEAD
```

プッシュが指示されていない場合はスキップする。

## 注意事項

- **必ず品質チェックを実施**してからコミット
- テストが失敗している場合は、関連するテストか確認
- エラーや警告は可能な限り解消してからコミット
- `git add -A` は使用禁止（関係ないファイルを含めないため）
- パスに特殊文字が含まれる場合は適切にエスケープ

## このスキルを使わないケース

以下のキーワードが含まれる場合は、このスキルではなく対応するスキルを使用すること:

| キーワード | 使用すべきスキル |
|-----------|----------------|
| 「ブランチ作成」「branch」「PRを作って」 | `branch` |
| 「マージ」「merge」「PRをマージして」 | `merge` |
| 「CI修正」「CIが落ちてる」 | `ci-fix` |
| 「QAチェック」「lint」「型チェック」 | `qa-check` |

このスキルは **既にステージング可能な変更がある状態で「コミットして」「pushして」と明示的に指示された場合のみ** 使用する。
