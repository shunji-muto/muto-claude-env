# Senior PR Review — 出力フォーマット

subagent はこの形式に**厳密に**従って出力する。SKILL.md から `Read` され、prompt に埋め込まれる。

---

## レポート形式

```markdown
# {対象} レビュー (#{PR番号}|{plan path}) — preset: {preset名}

## Blocking
- [指摘内容を簡潔に] (`path/to/file.ts:42` または `plan §4.2`)
  - **理由**: なぜマージ前に必ず修正が必要か (1–2 行)
  - **修正方針**: 具体的にどう直すか (file:line 単位)
- [指摘 2] (...)

## Improvements
- [指摘内容] (`path/to/file.ts:88`)
  - 採用任意。Phase X+1 持ち越し可
- ...

## Concerns
- [懸念] (`path/to/file.ts:120`)
  - **未確定理由**: 検証していないが疑わしい点 (1 行)
  - **検証方法**: どうすれば確証が取れるか
- ...

## Positives
- [良判断] (`path/to/file.ts:55`)
  - 再現すべきパターン / 計画書との整合がとれている点
- ...

## 総合判断
**[GO / 修正後 GO / NO-GO]** — {理由 1–2 行}
```

---

## 採点軸の定義

各指摘を **必ず以下のいずれか 1 つ** に分類する。複数該当する場合は最も重い区分に入れる。

### Blocking (マージ前必須修正)

以下のいずれかに当てはまる:
- **仕様逸脱**: 計画書 / 設計意図と異なる実装
- **セキュリティバグ**: 認可漏れ、injection、token leak、open redirect 等
- **ロジックバグ**: 実行すれば再現する誤り (cache key 不整合、partition pruning 不能、状態遷移の取りこぼし)
- **データ整合性破壊**: scope filter 漏れ、JOIN による row 倍増、NULL 未防御
- **データ漏洩**: 他店舗 / 他組織 / 他ユーザーのデータが見える
- **a11y 重大欠陥**: keyboard 操作不能、screen reader で識別不能、focus trap 不在の dialog
- **型安全性破綻**: `as any` で runtime error の温床になっている

### Improvements (軽微・任意)

- 命名 (より明確に)
- コメント追加 / TODO 整理
- エッジケース防御の追加
- 重複コードの抽出
- Phase X+1 / 後続 PR で対処可能

### Concerns (確証なしの懸念)

- 「動くと思うが負荷時の挙動が読めない」
- 「テストが無いので X のケースが回帰しないか不安」
- 「他箇所での同パターン使用と整合がとれているか未確認」
- → **検証方法を必ず併記**。Blocking に格上げ可能性のあるものはここに置く

### Positives (良判断)

- 計画書通りに実装されている
- 既存パターンを正しく踏襲している
- エラーハンドリングが適切
- テストが必要箇所をカバーしている
- 後続 PR で再利用しやすい設計になっている

---

## 必須ルール

1. **各指摘には根拠を付ける**: `file:line` または `plan §X.Y`。根拠なしの指摘は出力しないこと
2. **推測は Concerns へ**: 確信があるもののみ Blocking / Improvements。曖昧なら Concerns で「検証方法」を書く
3. **Positives を必ず 1 件以上挙げる**: レビューは粗探しではなく学習機会。良判断は再現すべきパターン
4. **総合判断を必ず最後に書く**: GO / 修正後 GO / NO-GO のいずれか
5. **コード修正は禁止**: Edit / Write は使わない。レポートのみ
